1 Product Overview

Fuel is a calorie and macro tracking app built on three principles: every screen has one clear purpose, every AI estimate is verified against authoritative databases before logging, and every meal can be logged in under 10 seconds.

Phase 1 Scope (What We're Building)

Text search against USDA FoodData Central + Open Food Facts · Barcode scanning for packaged foods · Manual entry of custom foods with personal food library · Dashboard with calorie ring and macro pills · Streak tracking · History with date range summaries · User auth (email + Google + Apple) · Profile & goals with auto-calculation via Mifflin-St Jeor

Phase 1 explicitly excludes: AI photo/voice logging, ghost meal predictions, confidence scoring, coaching insights, meal planning, recipe analysis, social features, and wearable integrations. Those are Phases 2–4.

2 Architecture Decisions

Every decision below was deliberated and locked in before development. Do not revisit unless there is a blocking technical reason.

Architecture pattern?
Monolith with clean package boundaries
Single Spring Boot app. Modules (auth, food, tracking, user) are Java packages with clear service interfaces. Can be extracted into microservices later. Avoids service discovery, inter-service comms, distributed tracing overhead.
Primary key format?
UUID across all tables
Universally unique, safe to expose in APIs, no sequential guessing.
Delete strategy?
Soft delete for user-facing data, hard delete for system data
Users, tracking entries, custom foods get deleted_at. Refresh tokens, verification codes are hard-deleted.
Auth strategy?
Email+password AND Google+Apple OAuth, with account linking
One user can have multiple auth methods. Stored in a separate user_auth_providers table.
Session expiry?
Sliding 30-day refresh token
Short access token (15 min) + long refresh (30 days). Each refresh resets the 30-day clock. Users stay logged in as long as they open the app within any 30-day window.
Email verification?
Yes, but allow app usage immediately
User gets email_verified: false and can use the app right away. Gets nudged to verify. Password reset requires verified email.
Food search strategy?
USDA first, Open Food Facts as fallback
USDA has higher quality data. If USDA returns <5 results, also query OFF and merge. Barcode lookups go to OFF first.
Food ID strategy?
Cache in local PostgreSQL with internal UUID
External API results are saved to our food_items table. Single ID space, resilient to API downtime, enables Redis caching.
Meal organization?
Optional meal labels (flat timeline + optional tags)
Entries are a timestamped flat list. Users CAN optionally tag as breakfast/lunch/dinner/snack.
Dashboard API?
Single endpoint returning everything
One call returns calorie totals, macro totals, entries, and streak. One request per screen load.
Daily totals?
Compute on the fly + Redis cache
SUM 10–20 rows with an indexed query takes <1ms. Redis caches the result. No separate daily_summaries table needed.
Goal setting?
Auto-calculate (Mifflin-St Jeor) with manual override
Body stats suggest goals. User can accept or override. Auto-recalculates when stats change (only if user hasn't overridden).

6 Auth Module

Token Strategy

TokenLifetimeStoragePurpose
Access Token (JWT)15 minutesClient-side only (mobile secure storage)Sent with every API request. Contains userId, email, iat, exp. Signed HS256.
Refresh Token30 days (sliding)SHA-256 hash in PostgreSQL. Raw token on client.Used only at /auth/refresh to get new access tokens. Each use resets the 30-day clock.
Security Rule: Never store raw refresh tokens in the database. Always store a SHA-256 hash. The raw token exists only on the client device and in transit over HTTPS.

• Password hashed with BCrypt cost factor 12 — balances security vs. speed.
• Default goals (2000 cal) are created so the dashboard has something to display immediately.
• Verification email is sent asynchronously — if the email service fails, registration still succeeds.
• accessTokenExpiresIn: 900 (seconds) and refreshTokenExpiresIn: 2592000 are included so the app knows when to refresh.

Account linking logic (3-step check):
1. Check if this Google/Apple sub is already in user_auth_providers → returning user, issue tokens.
2. Check if this email exists in users → existing email+password user, link the new OAuth provider to their account.
3. Neither found → create brand new user + provider.

Apple-specific: Apple only sends the user's name on the first sign-in. On subsequent logins, fullName is null. You must store it on first use or it's lost forever.

isNewUser tells the mobile app whether to show the onboarding flow (body stats, goal setting) or go straight to the dashboard.

    end

    AS->>RTR: delete(oldToken)
    AS->>AS: Generate new accessToken (fresh 15-min expiry)
    AS->>AS: Generate new refreshToken → SHA-256 hash
    AS->>RTR: save(userId, newTokenHash, expiresAt=now+30days)

Replay Attack Protection: If someone uses a refresh token that has already been rotated (not found in DB), this likely means a token was stolen and the attacker used it. The service MUST delete ALL refresh tokens for that user, forcing re-authentication on every device. This is a critical security measure.

8 Tracking Module

Critical Design Rule: Nutrition Snapshots. When a food is logged, the nutrition values (calories, protein, carbs, fat, fiber) are calculated and stored ON THE ENTRY. They are never recomputed from the food_items table. If USDA updates chicken breast from 165 to 170 cal, yesterday's log still shows 165. Historical accuracy is non-negotiable.

Snapshot Calculation Formula

ratio = entry.servingSize / food.servingSize
entry.calories = ROUND(food.calories × ratio × entry.quantity, 1)
entry.protein  = ROUND(food.protein  × ratio × entry.quantity, 2)
entry.carbs    = ROUND(food.carbs    × ratio × entry.quantity, 2)
entry.fat      = ROUND(food.fat      × ratio × entry.quantity, 2)
entry.fiber    = ROUND(food.fiber    × ratio × entry.quantity, 2)

Example:
  Food: Chicken breast — 100g serving, 165 cal, 31g protein, 0g carbs, 3.6g fat
  User logs: 150g, quantity = 1
  ratio = 150 / 100 = 1.5
  calories = 165 × 1.5 × 1 = 247.5
  protein  = 31  × 1.5 × 1 = 46.5
  carbs    = 0   × 1.5 × 1 = 0.0
  fat      = 3.6 × 1.5 × 1 = 5.4

11 Redis — Full Deep Dive

Why Redis? (And Why Not Something Else?)

We have two performance problems Redis solves:

Problem 1: External API Latency

When a user searches for "chicken breast," we call the USDA API (200–500ms round trip) and potentially Open Food Facts (300–800ms). Without caching, every search by every user hits these external APIs. With 1,000 users each searching 5 times a day, that's 5,000 external API calls — and USDA's rate limit is only 3,600/hour. We'd hit rate limits within the first hour and break search for everyone.

Redis solves this: The first user who searches "chicken breast" pays the 500ms API cost. Every subsequent user searching the same term gets the result from Redis in <1ms. The external API call never happens again until the cache expires (24 hours).

Problem 2: Dashboard Read Frequency

The dashboard is the most-viewed screen in the app. Users open it every time they want to check their progress — potentially 10–20 times per day. Each view requires: querying all tracking entries for the day, summing calories and macros, fetching user goals, and calculating the streak. While each of these queries is fast individually (<5ms), under load with many concurrent users, these repeated reads add up.

Redis solves this: After the first dashboard load, the computed result is cached. The next 9–19 views of the same day's dashboard come from Redis in <1ms with zero database load. The cache only invalidates when the user actually logs, edits, or deletes a food entry.

Why Redis specifically (and not an in-memory cache like Caffeine/Guava)?

  • Survives app restartsAn in-memory cache (Caffeine, Guava) lives inside the JVM. Every time you redeploy or restart the Spring Boot app, the cache is gone and every user's next request hits the database and external APIs cold. Redis persists across restarts because it's a separate process.
  • Shared across instancesIf you ever run 2+ instances of the app (load balancing), an in-memory cache would be per-instance — user A's search result cached on instance 1 wouldn't help user B on instance 2. Redis is shared, so one cache serves all instances. Not needed for Phase 1 (single instance), but avoids a future migration.
  • We already need it for rate limitingRate limiting (5 login attempts/min, 30 searches/min) requires a shared counter with TTL. Redis is the standard tool for this. Since we're already running Redis, using it for caching costs nothing extra.
  • Built-in TTL (time to live)Redis natively expires keys after a set duration. No custom cleanup code needed. Set a key with EX 86400 and it vanishes in 24 hours automatically.

Every Cache Key Explained

There are exactly 4 key patterns in Phase 1. Here's what each one stores, why it exists, and how its TTL was chosen.

food:search:{sha256(query:page:size)}
TTL: 24 hours Value type: JSON list of food item UUIDs Set by: FoodService.search() Invalidated by: Time-based only (TTL expiry)
What it stores: The list of food item IDs (UUIDs) that matched a particular search query. Not the full food data — just the IDs. The actual food data lives in PostgreSQL's food_items table and is fetched by ID on cache hit.

Why the key includes page and size: A search for "chicken breast" page 0 size 20 returns different results than page 1 size 20. The SHA-256 hash of "chicken breast:0:20" vs "chicken breast:1:20" produces different cache keys.

Why 24-hour TTL: Food databases (USDA, OFF) update infrequently — maybe weekly. 24 hours balances freshness with cache efficiency. A new food added to USDA will appear in search results within at most 24 hours.

Why no explicit invalidation: We don't control when USDA/OFF add or update foods. There's no event we can listen to. TTL-based expiry is the only practical option.

food:barcode:{barcode_number}
TTL: 7 days Value type: Single food item UUID Set by: FoodService.lookupBarcode() Invalidated by: Time-based only (TTL expiry)
What it stores: The internal UUID of the food item that matches this barcode. One barcode always maps to one product.

Why 7-day TTL (longer than search): Barcode-to-product mappings are extremely stable. Barcode 0023700011527 will be Tyson Chicken Strips today, next week, and next year. The nutrition data might update slightly, but the mapping doesn't change. 7 days minimizes unnecessary external API calls for an essentially static lookup.

Key: food:barcode:0023700011527
Value: "a23bc45d-67ef-8901-b234-567890abcdef"

food:detail:{foodId}
TTL: 48 hours Value type: Full food item JSON Set by: FoodService.getFoodById() Invalidated by: Time-based only. Also on custom food update/delete.
What it stores: The complete food item data as JSON — name, brand, source, serving info, all macros, alternative serving sizes. This avoids a PostgreSQL read when the user taps on a food item they've already viewed.

Why 48-hour TTL: Shorter than barcode (data can be more detailed and might update) but longer than search (it's a single item, not a result set). 48 hours is a middle ground.

When to explicitly invalidate: If a user updates their custom food (PUT /api/food/custom/{id}), delete food:detail:{id} so the next read picks up the changes.

dashboard:{userId}:{date}
TTL: 5 minutes Value type: Full dashboard response JSON Set by: DashboardService.getDashboard() Invalidated by: Any write to tracking_entries for this user+date. Any change to user_goals.
What it stores: The entire dashboard response — calorie totals, macro totals, entries list, streak data. This is the single most impactful cache because the dashboard is loaded 10–20 times per day per user.

Why 5-minute TTL (much shorter than food caches): This data changes frequently (every time the user logs a meal). The 5-minute TTL is a safety net, not the primary expiry mechanism. The real expiry is explicit invalidation — we delete the cache key every time a tracking entry is created, updated, or deleted. The 5-minute TTL only matters if the invalidation somehow fails (bug, race condition). Without it, a stale dashboard could persist indefinitely.

Why user-specific + date-specific: User A's dashboard for June 9 is completely different from User B's. And User A's June 9 dashboard is different from their June 8 dashboard. The compound key ensures isolation.

Example:

// Rule 1: Tracking entry created, updated, or deleted
// Invalidate the dashboard for THAT specific user + date only.
// Other dates and other users are unaffected.

public TrackingEntry logEntry(LogEntryRequest req, UUID userId) {
    TrackingEntry entry = // ... save to DB
    redisTemplate.delete("dashboard:" + userId + ":" + entry.getLogDate());
    return entry;
}

// Rule 2: User goals changed (manual override or recalculation)
// Invalidate ALL dashboard caches for this user, because goals affect
// the "remaining" calculation on every date's dashboard.

public UserGoals updateGoals(GoalsRequest req, UUID userId) {
    UserGoals goals = // ... save to DB
    Set<String> keys = redisTemplate.keys("dashboard:" + userId + ":*");
    if (keys != null && !keys.isEmpty()) {
        redisTemplate.delete(keys);
    }
    return goals;
}

// Rule 3: Custom food updated or deleted
// Invalidate the food detail cache for that specific food.
// Dashboard caches are NOT affected because tracking entries store
// nutrition snapshots — they don't reference food_items at read time.

public FoodItem updateCustomFood(UUID foodId, UpdateFoodRequest req) {
    FoodItem food = // ... save to DB
    redisTemplate.delete("food:detail:" + foodId);
    return food;
}

// Rule 4: Food search and barcode caches are NEVER explicitly invalidated.
// They expire naturally via TTL. This is intentional — we don't control
// when USDA/OFF update their data, so TTL-based expiry is the only option.

When Redis Goes Down

Redis is a cache, not a source of truth. If Redis is unavailable, the app must still work. Every query that goes through Redis must have a PostgreSQL fallback.

// Implementation pattern: cache-aside with fallback

public DashboardResponse getDashboard(UUID userId, LocalDate date) {
    String cacheKey = "dashboard:" + userId + ":" + date;

    // 1. TRY Redis
    try {
        String cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return objectMapper.readValue(cached, DashboardResponse.class);
        }
    } catch (RedisConnectionFailureException e) {
        log.warn("Redis unavailable, falling back to DB: {}", e.getMessage());
        // Don't rethrow — continue to DB
    }

    // 2. Compute from PostgreSQL (this ALWAYS works)
    DashboardResponse response = computeDashboardFromDb(userId, date);

    // 3. TRY to cache the result (best-effort)
    try {
        String json = objectMapper.writeValueAsString(response);
        redisTemplate.opsForValue().set(cacheKey, json, Duration.ofMinutes(5));
    } catch (RedisConnectionFailureException e) {
        log.warn("Failed to cache dashboard: {}", e.getMessage());
        // Don't rethrow — the user still gets their data
    }

    return response;
}
Critical rule: Redis errors must NEVER cause a user-facing error. Wrap every Redis call in a try-catch. If Redis throws, log a warning and fall back to PostgreSQL. The user should never know Redis exists.
